# Manager Knowledge Guide

这份文档定义 `AI经理知识库` 在 AgentHub 里的定位。

它的目标不是替代运行时工具，而是让 `AI经理` 对这些问题具备稳定认知：

- 平台是什么
- 角色边界是什么
- 新员工怎么接入
- 多设备 / 多员工模型怎么理解
- 后续怎么继续扩经理职责

## 为什么要单独做知识库

如果只靠 prompt，经理会有三个问题：

1. 知识和行为混在一起，不好维护
2. 模型一换，知识容易丢
3. 本地回退模式几乎没法复用这层认知

所以现在 AgentHub 把经理知识拆成了独立目录：

- `knowledge/manager/*.md`

运行时会在服务启动时加载这些条目，并提供 `search_manager_knowledge` 给经理调用。

本地回退模式也会复用同一份知识，而不是再维护一套平行解释。

## 什么内容该放知识库

适合放进知识库的内容：

- 平台规则
- 接入方式
- 术语解释
- 角色边界
- 推荐工作流
- 常见问答

不适合放进知识库的内容：

- 实时在线状态
- 当前任务进度
- 待审批列表
- 当前工作区清单
- 谁卡住了 / 谁掉线了 / 谁久未更新
- 给员工发催办或纠偏消息

这些都属于运行时状态，应该通过经理工具获取。

## 一条简单判断规则

- `稳定事实` -> 知识库
- `实时状态` -> 工具
- `行为变化` -> 协议 / 代码 / 文档

例如：

- “别的 Agent 怎么接入这个平台” -> 知识库
- “Codex Main 现在在做什么” -> 工具
- “现在最该关注什么” -> 工具
- “催一下 Codex Main，让他汇报进度” -> 工具
- “支持群协作任务和多员工会签” -> 很可能需要协议变更

## 知识条目格式

每个知识文件建议使用带元数据的 Markdown：

```md
---
id: agent-onboarding
title: 新 Agent / 新设备接入 AgentHub
summary: 说明新员工如何注册到 Hub 并声明工作区。
keywords: 接入, onboarding, 新设备, codex
---

# 新 Agent / 新设备接入 AgentHub

这里写条目正文。
```

字段含义：

- `id`
  稳定知识 ID
- `title`
  给经理和开发者看的主题名
- `summary`
  检索命中后优先给经理的简版摘要
- `keywords`
  方便经理按自然语言问题召回

## 当前知识目录

第一版建议至少维护这些主题：

1. `manager-responsibilities`
2. `platform-model`
3. `agent-onboarding`
4. `extend-manager`

这样经理至少能回答：

- 你是谁
- 你能做什么
- 其他 Agent 怎么接入
- 这个平台的模型是什么
- 后续怎么扩知识和职责

## 后续怎么扩经理职责

推荐顺序：

1. 先判断是不是纯知识问题
2. 如果是，新增或更新 `knowledge/manager/*.md`
3. 如果不是，再判断是不是需要新工具
4. 如果会改执行模型，再更新协议和架构文档

不要一上来就把新能力硬写进 prompt 或 UI。

## 开发约束

扩经理能力时，优先保证这三件事：

1. 人类仍然通过“说目标”来驱动系统
2. 经理只在必要时暴露系统结构
3. 实时事实永远由工具确认，不由知识库假装生成
