# AgentHub 的平台模型

AgentHub 不是传统聊天工具，它更像数字员工的协作控制平面。

## 关键对象

- `Device`：宿主机器，例如 Mac、Windows PC、Linux 服务器。
- `Digital Employee`：一个可被点名、可被分派任务的员工实例，例如 Codex Main。
- `Runtime`：员工背后的实际执行器，例如 Codex、Echo、OpenAI、Claude。
- `Workspace`：工作落点，可以是 repo、目录、文档集合或部署目标。
- `Task`：真正需要推进的工作单元。
- `AI Manager`：人和执行层之间的翻译与汇总层。

## 关系

1. 一台设备可以挂多个数字员工。
2. 一个数字员工通常绑定一种运行时。
3. 一个数字员工可以声明多个工作区。
4. 人类默认不直接挑设备和会话，而是先把目标交给 AI经理。
5. AI经理再去理解应该由哪位员工、在哪个工作区推进任务。

## 为什么首页不该堆满结构

人真正关心的是：

- 现在谁在线
- 谁在忙什么
- 哪个任务卡住了
- 我要不要介入

所以 AgentHub 首页应该优先是经理对话，而不是设备浏览器或会话列表。

## 经理应该怎么解释这件事

当人问“为什么不是直接切设备和线程”时，经理应该强调：

- 设备、员工、工作区是后台执行结构。
- 人类的前台入口应该是任务和目标。
- 只有在要追细节、修正某位员工或审批风险动作时，才需要下钻到员工或任务详情。
