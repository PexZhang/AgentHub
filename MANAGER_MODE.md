# Manager Mode

这份文档描述的是 `人类管理台` 的交互，不是整个产品的唯一中心。

新的前提是：

- `Agent` 才是平台的一线使用者
- `AI经理` 是人和执行层之间的翻译与汇总层
- `人类页面` 只负责观察、指挥、授权和下钻

所以这里定义的是 `thin human console`，不是 `thick human control center`。

## 问题定义

当前版本更像一个远程控制台：

- 用户要先理解设备、Agent、会话这些内部结构
- 高密度信息适合桌面，不适合手机
- 系统把“执行结构”暴露成了“用户入口”

但用户真实诉求是：

- 我现在想推进什么任务
- 现在谁在忙
- 某个员工现在做到哪一步了
- 我想直接指导某个员工

所以新版本要把 `系统结构` 降级成 `后台执行层`，把 `经理式任务协作` 升级成 `前台交互层`。

但更进一步地说，AgentHub 的真正产品中心不该是这个经理页面，而该是 `Agent 执行与协作协议`。经理页面只是给人看的薄壳。

## 新的产品心智

AgentHub 应该被理解为：

- `AI经理`：唯一默认入口，负责理解任务、分派员工、汇总结果
- `员工`：分布在不同设备上的执行者
- `工作区`：任务落点，可以是 repo、目录、文档空间或部署路径
- `任务`：用户真正关心的对象

关系是：

`用户 -> AI经理 -> 工作区 / 员工 -> 任务结果`

而不是：

`用户 -> 设备 -> Agent -> 会话`

## 默认交互

首页默认只有一个主问题：

`你现在想推进什么任务？`

用户可以直接说：

- `现在我的 agent 员工有哪些`
- `他们现在在做什么`
- `帮我看看 XXX 在做啥，进度咋样了`
- `帮我切到和 XXX 的对话，我要指导他`
- `去 Office PC 上整理 docs 目录`
- `帮我看下 Linux 机器上的支付服务日志`

`AI经理` 负责：

- 识别任务意图
- 定位工作区
- 判断应该交给哪位员工
- 汇总当前进度
- 必要时发起最小确认

## 一级页面

新的 IA 建议只保留 3 个一级入口：

1. `经理`
默认首页。用户只和 AI 经理说话。

2. `任务`
查看执行中的任务、待确认事项、已完成结果。

3. `员工`
低频管理页，用来查看员工、设备、能力和在线状态。

说明：

- `设备` 不再是一级入口
- `工作区` 不再强制常驻，而是作为任务上下文出现
- `直连会话` 不是一级入口，而是从经理页或任务页里临时进入

补充：

- `指挥总览` 可以作为一个低频监督页存在，但它不是新的首页
- 它的作用不是替代经理，而是只在你想“看全局、抓异常、做介入”时快速展开
- 首页仍然应该是 `AI经理对话`

## 经理页

经理页是最重要的人类页面，但不是最重要的产品对象。

它应该包含 4 个区块：

1. `经理输入区`
一个强输入框，默认文案是：
`告诉我你现在想推进什么任务`

2. `经理汇总`
一屏内只展示最关键的运营信息，例如：

- 当前在线员工数
- 执行中的任务数
- 等待你确认的任务数
- 最近阻塞的任务

3. `经理对话流`
不是普通聊天泡泡，而是“经理汇报流”：

- 用户提问
- AI 经理给出结构化总结
- 必要时附带任务卡片和员工卡片

4. `快速动作`
不是让用户配置系统，而是让用户快速发起经理式提问：

- `谁在线`
- `谁最忙`
- `谁卡住了`
- `查看待确认`

## 任务页

任务页展示的是 `任务`，不是 `线程`。

一张任务卡应该包含：

- 任务标题
- 当前状态
- 当前负责人
- 当前工作区
- 最近更新时间
- 最近一条关键进展
- 是否需要人工确认

任务状态建议统一为：

- `待分派`
- `处理中`
- `等待确认`
- `已完成`
- `失败`

任务页提供两个关键动作：

- `查看进度`
- `切到直连`

## 员工页

员工页是低频管理入口，不是默认首页。

它展示：

- 员工名
- 所在设备
- 运行时
- 在线状态
- 当前任务
- 最近心跳
- 能力标签

用户在这里主要做 3 件事：

- 看当前有哪些员工
- 看每个员工在忙什么
- 主动切到与某个员工的直连对话

## 直连模式

直连模式是二级模式，不应替代经理模式。

触发方式：

- 用户说：`帮我切到和 XXX 的对话`
- 用户在任务卡片里点击 `切到直连`
- 用户在员工页里点击 `直连`

进入直连模式后，界面要明确显示：

- 当前直连的是谁
- 他所在的设备
- 他当前正在做的任务
- 一个明显的 `返回AI经理` 入口

直连模式的作用是：

- 临时细化指令
- 处理阻塞
- 给单个员工补充上下文

它不是默认工作模式。

## 工作区的展示原则

工作区仍然很重要，但不该先于任务出现。

工作区只在这 3 种场景露出：

1. AI经理已经帮你完成路由之后
例如：
`已定位到 Office PC / docs 工作区`

2. AI经理不确定任务范围时
例如：
`你是要我处理 Mac 上的 AgentHub 仓库，还是 Windows 上的测试目录？`

3. 用户主动进入员工页或资源管理页

## AI经理的最小确认规则

AI经理默认不要把选择题甩给用户。

只有以下情况才追问：

1. 工作区歧义明显
2. 风险操作需要授权
3. 当前任务信息不足，无法启动

除这 3 种外，默认直接路由并执行。

## 手机端原则

手机端要遵守 5 条原则：

1. 首页只做一件事：说任务
2. 默认先看总结，不先看配置
3. 大结构隐藏，必要时再展开
4. 任务卡优先于系统卡
5. 直连是临时模式，不是主模式

## 人类指挥官的重要场景

有些场景用户不一定会主动提，但系统应该帮他准备好：

1. `待审批`
某位员工已经走到风险动作前，只差人类拍板。

2. `任务阻塞`
员工没有继续推进，且已经明确给出阻塞原因。

3. `长时间无更新`
任务还处于执行中，但已经很久没有新进展，需要判断是正常长耗时还是已经停住。

4. `员工离线但任务未结束`
某台设备掉线、休眠或 Agent 退出了，但任务还挂在执行态。

5. `跨设备查看同一份状态`
用户换了手机、平板或另一台电脑后，应该读到同一份任务与员工快照，而不是每台设备看到不同的局部状态。

6. `直连纠偏`
用户不需要长期处在直连模式里，但在某位员工偏航、停住或需要补充上下文时，应该能一跳进入。

7. `未来多员工群讨论`
当一个任务需要多个员工共同对齐方案时，系统应该支持从经理视角组织多人讨论，而不是要求用户逐个切换单聊。

这些场景说明：

- `人类页面` 不该展示所有结构
- 但必须主动暴露 `需要人处理的异常`
- 并且提供 `回经理处理` 与 `进入直连` 两种纠偏入口

## 指挥总览页的边界

如果提供 `指挥总览`，它应该只负责三件事：

1. `态势摘要`
在线员工、执行中任务、待审批、阻塞数。

2. `需要关注`
把阻塞、待审批、长时间无更新、离线未结束任务直接提出来。

3. `员工态势`
每位员工只展示最常用的监督字段：

- 在哪台设备
- 当前任务
- 当前工作区
- 最近同步时间
- 简短进展摘要

它不应该变成：

- 配置中心
- 设备浏览器
- 复杂看板
- 多层筛选器首页

## 关键示例

### 示例 1：问员工名单

用户：
`现在我的 agent 员工有哪些`

AI经理：
`当前有 6 位员工在线，2 位离线。MacBook 上有 Codex Main 和 Claude Review；Office PC 上有 Office Codex；Linux 上有 Deploy Agent 和 Log Agent。`

### 示例 2：问全员状态

用户：
`他们现在在做什么`

AI经理：
`Codex Main 正在修改 AgentHub 的移动端页面；Office Codex 正在整理 docs 目录；Deploy Agent 当前空闲；Claude Review 正在等待你确认评审建议。`

### 示例 3：问单个员工进度

用户：
`帮我看看 Office Codex 在做啥，进度咋样了`

AI经理：
`Office Codex 当前正在处理“整理招标材料”任务，已经完成目录归类和命名检查，下一步是合并重复文件，目前没有阻塞。`

### 示例 4：切到直连

用户：
`帮我切到和 Codex Main 的对话，我要指导他`

AI经理：
`已切到和 Codex Main 的直连对话。当前他正在处理“AgentHub 移动端交互调整”。`

## 实施建议

建议分 3 步落地：

1. `先做经理模式 UI`
- 首页改成经理入口
- 员工和任务降到二级视图
- 直连模式做成明确的次级态

2. `再补经理层能力`
- 员工盘点
- 任务汇总
- 指定员工切换
- 任务进度总结

3. `最后做自动路由`
- 任务自动匹配工作区
- 自动选员工
- 多员工协作与结果汇总

## 结论

AgentHub 不应该继续优化成一个更复杂的控制台。

它应该变成：

`一个面向 Agent 的协作控制平面，加上一个薄的 AI经理观察台，替人调度数字员工并推进任务。`
